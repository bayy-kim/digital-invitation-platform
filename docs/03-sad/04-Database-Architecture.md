Database OverviewPlatform SaaS Undangan Digital ini mengadopsi arsitektur pangkalan data hibrida untuk memisahkan beban kerja transaksional intensif dari kebutuhan pembacaan statis berkecepatan tinggi:Pangkalan Data Utama (PostgreSQL 16): Berperan sebagai satu-satunya sumber kebenaran (single source of truth) data transaksional terstruktur platform. PostgreSQL dipilih karena kepatuhan penuh terhadap standar ACID, dukungan native tipe data JSONB berkinerja tinggi, pengindeksan tingkat lanjut (GIN, GiST), serta keandalan skalabilitas read-replica.Pangkalan Data In-Memory \& Cache (Redis v7): Bertindak sebagai lapisan akselerasi memori untuk mereduksi kueri PostgreSQL secara langsung. Redis mengelola session token, throttling rate limiter, caching data undangan aktif (Active), serta bertindak sebagai broker antrean transaksional (message queue broker).2. Multi-tenancy StrategyPlatform ini mengadopsi model Logical Data Isolation di dalam pangkalan data tunggal (Shared Database, Shared Schema):Mekanisme Isolasi: Semua baris data transaksional diisolasi menggunakan kunci kolom user\_id atau reseller\_id.Row-Level Security (RLS): Di tingkat PostgreSQL, kebijakan RLS diaktifkan pada tabel-tabel sensitif seperti orders, invitations, dan wallets. Setiap sesi kueri yang diinisiasi oleh backend Laravel wajib menyuntikkan konteks ID pengguna terautentikasi ke dalam variabel sesi PostgreSQL (SET LOCAL app.current\_user\_id = '...') sehingga sistem database secara otomatis memfilter data tanpa bergantung sepenuhnya pada penulisan klausa WHERE di tingkat kode aplikasi.4. Primary Key StrategySemua tabel transaksional dan master (kecuali log murni berkinerja tinggi seperti invitation\_analytics) wajib mengadopsi skema UUIDv4 (Universal Unique Identifier) sebagai Primary Key:Alasan Keamanan: UUID menyembunyikan ukuran/volume transaksi riil platform dari kompetitor dan mencegah eksploitasi perayap (web scraping) berbasis tebakan nomor urut incremental ID (Insecure Direct Object Reference).Alasan Skalabilitas: UUID dapat di-generate langsung secara aman di sisi Backend Laravel sebelum menyentuh database, mempermudah partisi database (sharding) atau migrasi multi-cluster di masa depan tanpa terjadinya konflik duplikasi kunci utama.7. Foreign Key StrategySemua hubungan relasi antar tabel wajib dideklarasikan secara eksplisit menggunakan batasan kunci asing (FOREIGN KEY constraints):Pencegahan Data Yatim (Orphaned Data): Menjamin integritas referensi pangkalan data di tingkat mesin, bukan sekadar di tingkat logika framework aplikasi.Eksplisit Indexing: Setiap kolom Foreign Key wajib dipasangkan indeks B-Tree secara eksplisit di database karena PostgreSQL tidak membuat indeks pada kolom FK secara otomatis. Tanpa indeks ini, operasi JOIN atau penghapusan data relasional akan melambat drastis karena PostgreSQL dipaksa melakukan scanning tabel penuh (sequential table scan).8. Index StrategyOptimalisasi kueri dilakukan dengan menempatkan jenis indeks PostgreSQL yang paling sesuai pada skenario kueri yang sering dijalankan:Indeks B-Tree (Default): Digunakan untuk perbandingan rentang (>, <), pencarian setara (=), dan sorting data. Dipasang pada semua Primary Key (UUID), Foreign Key, kolom slug (invitations.slug, templates.slug), dan kolom status.Indeks GIN (Generalized Inverted Index): Khusus digunakan untuk mengindeks struktur data JSONB di tabel invitation\_drafts dan penelusuran teks pencarian katalog penuh (Full-Text Search).9. Composite Index StrategyIndeks komposit (indeks multi-kolom) diterapkan untuk mengoptimalkan kueri kompleks yang menggunakan kombinasi klausa WHERE dan ORDER BY secara berulang:idx\_rsvps\_invitation\_attendance:Komposit B-Tree pada tabel guest\_rsvps(invitation\_id, attendance). Mempercepat kalkulasi total kehadiran tamu dinamis pada diagram dasbor.idx\_analytics\_invitation\_date:Komposit B-Tree pada tabel invitation\_analytics(invitation\_id, viewed\_at). Mengoptimalkan kueri penarikan grafik garis analitik rentang tanggal tertentu.10. JSONB StrategyStruktur desain visual undangan yang diolah lewat Builder disimpan sebagai dokumen JSONB pada tabel invitation\_drafts:Alasan Pemilihan JSONB: Berbeda dengan tipe data JSON biasa yang menyimpan dalam bentuk teks biasa, JSONB menyimpan data dalam bentuk biner terurai (deconstructed binary). Proses tulis sedikit lebih lambat, namun proses baca dan manipulasi properti di dalamnya berjalan jauh lebih cepat.Indeksasi GIN:GIN Index dipasang pada kolom draft\_payload untuk mendukung pencarian properti desain dinamis secara instan:CREATE INDEX idx\_draft\_payload\_gin ON invitation\_drafts USING gin (draft\_payload);JSONB Querying: Backend Laravel dapat langsung memodifikasi nilai properti spesifik di dalam payload (misal: hanya memperbarui warna tema utama) di tingkat database menggunakan fungsi kustom jsonb\_set tanpa perlu mendownload dan menyimpan ulang dokumen JSONB utuh dari server.11. Native UUID StrategyPenyimpanan UUID dilakukan menggunakan tipe data native UUID di PostgreSQL, bukan dalam format string panjang (VARCHAR(36)):Efisiensi Storage: Native UUID PostgreSQL hanya memakan memori sebesar 16 bytes penyimpanan, bandingkan dengan format representasi teks VARCHAR yang memakan memori sebesar 36 bytes.Kecepatan Indeks: Ukuran penyimpanan yang ringkas menghemat ruang memori RAM (buffer cache) saat PostgreSQL melacak indeks pencarian B-Tree, mempercepat respon kueri tulis dan baca hingga 50%.12. Audit ColumnsSetiap tabel master dan transaksional yang dikelola secara langsung oleh admin maupun kustomer wajib menyertakan kolom audit standar untuk keperluan audit investigatif internal (audit trail):created\_by: UUID, NULLABLE (mengacu ke users.id). Mencatat aktor yang menginisiasi pembuatan data.updated\_by: UUID, NULLABLE (mengacu ke users.id). Mencatat aktor yang melakukan modifikasi terakhir.deleted\_by: UUID, NULLABLE (mengacu ke users.id). Mencatat aktor yang melakukan proses penghapusan (Soft Delete).13. Soft Delete StrategyMenghapus data kustomer secara permanen dari pangkalan data merupakan praktik berbahaya di lingkungan SaaS produksi. Sistem mengadopsi mekanisme Soft Delete:Logical Flag: Menggunakan kolom deleted\_at dengan tipe data TIMESTAMPTZ yang bersifat NULLABLE.Indeksasi Parsial (Partial Index): Untuk menghindari penurunan kecepatan pencarian kueri akibat menumpuknya data sampah yang sudah di-soft-delete, kueri harian kustomer menyaring data menggunakan indeks parsial:CREATE INDEX idx\_invitations\_active\_slug ON invitations (slug) WHERE deleted\_at IS NULL;Siklus Pembersihan (Database Garbage Collector): Task Scheduler Laravel dikonfigurasi untuk mengeksekusi perintah asinkron harian guna menghapus secara fisik (hard delete) data dengan tag Soft Delete yang sudah melampaui batas retensi hukum sistem (misal: lebih dari 90 hari).14. Timestamp StrategySeluruh pencatatan waktu di tingkat pangkalan data wajib mematuhi standar internasional ISO 8601 menggunakan tipe data TIMESTAMPTZ (timestamp with time zone):Alasan Pemilihan: Tipe data TIMESTAMP biasa tanpa zona waktu dapat memicu kekacauan fatal kalkulasi penayangan hitung mundur atau waktu akad nikah kustomer yang berada di wilayah zona waktu berbeda (WIB, WITA, WIT).Internasionalisasi: PostgreSQL secara internal mengonversi semua waktu masuk ke dalam format koordinat waktu universal UTC (Greenwich Mean Time) sebelum menyimpannya ke disk. Saat penarikan data, Postgres menyajikan kembali waktu tersebut sesuai dengan parameter zona waktu koneksi aktif sesi pengguna (session time zone).15. Database Naming ConventionKonsistensi penamaan wajib dijaga di seluruh migrasi schema demi mempermudah pemahaman tim backend developer:Nama Tabel: Menggunakan huruf kecil, kata dipisahkan tanda hubung bawah (snake\_case), dan berbentuk jamak (plural). Contoh: users, invitations, order\_items.Nama Kolom: Menggunakan format snake\_case tunggal (singular). Contoh: first\_name, expired\_at, category\_id.Penamaan Kunci Asing (Foreign Key): Menggunakan pola nama\_tabel\_tunggal\_id. Contoh: user\_id (mengacu ke tabel users kolom id).Penamaan Indeks: Mengikuti konvensi otomatis: idx\_\[nama\_tabel]\_\[nama\_kolom\_1]\_\[nama\_kolom\_2]. Contoh: idx\_guest\_rsvps\_invitation\_attendance.16. Database Constraints (Integritas Data Sisi Engine)Validasi tidak boleh dipercayakan 100% pada logika kode Laravel. Database harus bertindak sebagai benteng pertahanan terakhir integritas data melalui deklarasi batasan (constraints):Not Null Constraints: Dipasang di semua kolom kritis untuk mencegah kebocoran data rusak/tidak lengkap ke dalam database.Check Constraints (Pemeriksaan Logika):chk\_orders\_amounts: Memastikan total diskon tidak boleh melebihi nilai subtotal: CHECK (discount\_amount <= subtotal).chk\_rsvps\_pax: Memastikan jumlah pax yang didaftarkan tamu bernilai positif dan logis: CHECK (total\_persons >= 1 AND total\_persons <= 10).Unique Constraints: Mencegah terjadinya duplikasi pembuatan slug undangan publik aktif yang sama: UNIQUE (slug, deleted\_at).17. Cascade Rules \& Referential ActionsAturan penanganan relasi penghapusan data diatur secara ketat berdasarkan klasifikasi sensitivitas data:ON DELETE RESTRICT (Aturan Finansial): Dipasang pada tabel transaksi keuangan seperti orders dan payments. User dilarang keras dihapus dari database selama masih memiliki rekam jejak histori invoice transaksi finansial.ON DELETE CASCADE (Aturan Metadata): Dipasang pada sub-tabel pelengkap data draf undangan, seperti events, stories, galleries, invitation\_settings. Jika entitas induk invitations dihapus, seluruh data dekorasi seketika dibersihkan dari database secara otomatis guna mencegah penumpukan baris sampah tak berinduk.18. Transaction Strategy (ACID Compliance)Setiap operasi penulisan database yang melibatkan lebih dari dua tabel transaksional wajib dibungkus di dalam satu blok transaksi database tunggal:All-or-Nothing Execution: Menggunakan instruksi BEGIN, COMMIT, dan ROLLBACK. Contoh: saat kustomer mengklik tombol "Publish", pemindahan data dari payload draf JSONB ke baris tabel terpisah (events, stories) dan pencatatan log histori diproses di dalam satu siklus transaksi yang sama. Jika salah satu operasi gagal (misal: koneksi terputus saat menyimpan foto galeri), seluruh operasi dibatalkan total tanpa merusak data live sebelumnya.19. Database Locking StrategyUntuk menjaga keakuratan alokasi mutasi saldo kredit reseller atau status pembayaran saat trafik tinggi, database diatur menggunakan skema penguncian baris eksplisit:SELECT FOR UPDATE (Pessimistic Write Lock):Ketika menerima callback webhook pelunasan order atau penarikan dana reseller, sistem mengunci baris order atau baris saldo dompet terkait agar tidak dapat diakses atau dimodifikasi oleh kueri proses paralel lain selama siklus transaksi aktif berjalan. Hal ini memotong habis potensi bug celah pembobolan saldo ganda (double spending).20. Optimistic LockingUntuk operasi editor kolaboratif di Builder atau saat pengguna membuka draf edit undangan yang sama di dua perangkat atau tab browser berbeda, platform mengadopsi pola Optimistic Locking:Kolom lock\_version: Pada tabel invitation\_drafts, setiap draf menyertakan versi integer.Mekanisme Update: Setiap kali Laravel melakukan auto-save draf, query dijalankan dengan menyertakan versi terakhir yang diketahui:UPDATE invitation\_drafts SET draft\_payload = ?, lock\_version = lock\_version + 1 WHERE invitation\_id = ? AND lock\_version = ?;Pencegahan Konflik: Jika versi di database sudah berubah lebih tinggi (karena sudah di-save oleh tab browser lain), query update di atas akan mengembalikan total efek baris bernilai 0. Laravel mendeteksi ini, membatalkan transaksi, dan melemparkan respons peringatan konflik data kepada pengguna.21. Query Optimization \& Execution PlansAturan kueri database yang wajib ditaati oleh tim pengembang:Eager Loading Obligation: Melarang keras penggunaan kueri Lazy Loading Eloquent (Laravel) yang memicu masalah performa kueri berantai N+1. Semua kueri yang memuat relasi relasional wajib menggunakan instruksi deklarasi with() di awal.Index-Only Scans: Menyusun susunan kolom kueri seefisien mungkin. Hindari penulisan kueri seleksi liar SELECT \*. Cukup deklarasikan kolom spesifik yang dibutuhkan agar PostgreSQL dapat langsung mengembalikan data melalui pembacaan data indeks saja tanpa membuang memori RAM membaca baris data fisik di disk.22. Materialized Views (Analitik Super Admin)Kueri analitik finansial global (seperti perhitungan pendapatan berjalan bulanan - MRR, nilai umur pelanggan - LTV, dan persentase pembatalan - Churn Rate) sangat membebani komputasi database jika dihitung dari kueri mentah miliaran baris transaksi riil:Materialized View (mv\_financial\_metrics): Menyimpan hasil kalkulasi komputasi kueri analitik berat tersebut ke dalam tabel visual bayangan sementara di database.Siklus Refresh Otomatis: Scheduler Laravel dikonfigurasi untuk memicu pembaruan data secara berkala (misal: setiap jam atau tengah malam) secara asinkron di latar belakang tanpa mengganggu kinerja operasional transaksi API harian:REFRESH MATERIALIZED VIEW CONCURRENTLY mv\_financial\_metrics;23. Partitioning Strategy (Skalabilitas Skala Besar)Tabel log analitik kunjungan undangan publik (invitation\_analytics) berpotensi tumbuh masif hingga ratusan juta baris dalam hitungan bulan:Table Partitioning (Range Partitioning): Tabel invitation\_analytics dipecah secara fisik menjadi sub-tabel berdasarkan rentang waktu bulan berjalan, contoh:invitation\_analytics\_y2026m06 (Menyimpan data log bulan Juni 2026).invitation\_analytics\_y2026m07 (Menyimpan data log bulan Juli 2026).Alasan Performa: Saat admin menarik grafik tren bulanan kustomer, PostgreSQL secara cerdas hanya memindai sub-tabel partisi bulan terkait (partition pruning), mengabaikan miliaran baris log dari tahun atau bulan lalu, mempertahankan respons grafik analitik dasbor di bawah 1 detik.24. Full-Text Search (Pencarian Katalog)Pencarian katalog template di sisi depan (storefront) tidak menggunakan kueri lambat wildcard LIKE '%keyword%' yang menonaktifkan fungsi indeks database:PostgreSQL Full-Text Search Engine: Kolom pencarian dikonversi menjadi format tipe data pencarian teks vektor khusus (tsvector).GIN Index for Search: Indeks GIN khusus dipasang pada kolom representasi nama, deskripsi, dan tag template guna menyajikan pencarian teks multibahasa yang cepat, akurat, dan toleran terhadap kesalahan penulisan (typo tolerant).25. Backup Strategy (Rencana Cadangan Data)Keamanan data kustomer dari potensi bencana kerusakan fisik server dikawal dengan skema perlindungan ganda:Continuous Archiving (WAL Archiving): Mengaktifkan pengiriman log riwayat penulisan data transaksional (Write-Ahead Logging) PostgreSQL secara terus-menerus ke penyimpanan Cloud Object Storage (S3 compatible) terpisah menggunakan tool pgBackRest. Ini mendukung pemulihan data titik waktu presisi detik tertentu (Point-in-Time Recovery - PITR).Daily Logical Dump: Pengecekan otomatis harian (pukul 02:00 UTC) untuk membuat berkas backup skema dan baris data utuh terkompresi .sql.gz ke server cadangan fisik berbeda wilayah geografis (Offsite Backup).26. Restore Strategy \& Disaster RecoveryRencana pemulihan data diuji coba secara berkala untuk menjamin kesiapan tim infrastruktur saat terjadi kegagalan server total:RPO (Recovery Point Objective): Ditargetkan maksimum 5 menit. Artinya, jika server utama terbakar fisik, data yang dipulihkan dari WAL archive S3 dijamin tidak kehilangan riwayat transaksi lebih dari 5 menit sebelum bencana terjadi.RTO (Recovery Time Objective): Ditargetkan maksimum 30 menit untuk melakukan failover penuh mengalihkan trafik backend API ke server cadangan database replika baru.27. Zero-Downtime Migration RulesSetiap perubahan skema database (migrasi) di lingkungan produksi dilarang keras mengunci tabel operasional utama (avoid heavy locks) yang memicu matinya layanan:Adding Columns: Kolom baru yang ditambahkan wajib bersifat NULLABLE atau memiliki nilai default yang dideklarasikan secara asinkron.Creating Indexes: Pembuatan indeks baru di tabel produksi wajib menggunakan klausa CONCURRENTLY:CREATE INDEX CONCURRENTLY idx\_custom\_column ON table\_name(column);Ini memerintahkan PostgreSQL untuk membuat indeks di latar belakang tanpa menahan proses kueri tulis dan baca dari pengguna dasbor.28. Idempotent Seeder RulesBerkas pengisian data master database (Seeders di Laravel) wajib ditulis dengan prinsip Idempoten (dapat dijalankan berulang kali tanpa memicu duplikasi data eror):Larang Penggunaan insert() Mentah: Gunakan metode pengecekan cerdas seperti updateOrCreate() atau firstOrCreate() untuk mendistribusikan data master (seperti data peran, pilihan tema visual, warna default, dan konfigurasi sistem global) saat pertama kali melakukan inisialisasi instalasi server produksi atau pembaruan modul.29. Database Factory RulesUntuk penulisan modul pengujian otomatis (Unit \& Integration Testing):Strict Seed Factories: Factory wajib dideklarasikan menggunakan data UUID acak yang valid, menghindari penggunaan hardcoded ID.Transient States: Setiap pengujian fungsional berjalan di dalam siklus transaksi database (use RefreshDatabase / DatabaseTransactions traits) yang mengisolasi database pengujian kueri, menjaga agar database testing tetap bersih pasca pengujian diselesaikan.30. Connection Pooling (PgBouncer)Konkurensi koneksi pangkalan data dalam jumlah masif dari container-container Laravel dapat menghabiskan alokasi RAM server database dengan cepat (karena satu koneksi native PostgreSQL memakan konsumsi memori sekitar 10MB RAM):PgBouncer Integration: Bertindak sebagai proxy connection pooler ringan di depan server PostgreSQL. PgBouncer menahan ratusan koneksi dari backend Laravel ke dalam antrean pool dinamis, lalu mendistribusikannya ke dalam hanya puluhan koneksi aktif fisik nyata ke mesin PostgreSQL, menghemat konsumsi penggunaan memori RAM hingga 80%.31. Database Security ConfigurationNetwork Isolation (VPC): Server PostgreSQL diisolasi di dalam jaringan privat virtual (VPC) dan tidak memiliki IP publik terbuka. Akses masuk hanya diperkenankan dari subnet alamat IP container backend Laravel dan bastion host administrator via SSH tunneling.Encryption-at-Rest: Mengaktifkan enkripsi disk penyimpanan fisik server database (AWS KMS / LUKS Encryption) untuk melindungi data sensitif kustomer dari pencurian fisik hard disk server.SSL/TLS Connections Only: Mengonfigurasi file pg\_hba.conf untuk memaksa penolakan semua koneksi masuk yang tidak menggunakan enkripsi TLS (force ssl connection).32. Best Practices (Aturan Emas Tim Database)Larang Kueri Seleksi Liar (SELECT \*): Selalu sebutkan secara spesifik nama kolom yang dibutuhkan demi efisiensi transfer data memori.Kunci Nilai Keuangan: Data histori harga pada tabel order\_items wajib disalin secara independen saat pesanan dibuat, mengisolasi data riwayat transaksi dari perubahan harga template global di masa mendatang.Optimasi Pembersihan Vakum (Autovacuum Tuning): Konfigurasi daemon autovacuum pangkalan data diaktifkan secara berkala untuk membersihkan sisa-sisa baris mati (dead tuples) akibat masifnya proses auto-save Builder draf JSONB, mencegah terjadinya pembengkakan ukuran disk database (table bloat).33. KesimpulanCetak biru Database Architecture Specification ini menyajikan rancangan skema pangkalan data yang andal, aman, terindeks secara presisi, serta memiliki ketahanan skalabilitas kelas dunia. Melalui implementasi pemisahan status draf JSONB, row-level security, partisi log analitik, dan perlindungan isolasi jaringan VPC yang ketat, arsitektur database PostgreSQL 16 ini siap mendukung tim pengembang Anda dalam menghadirkan sistem SaaS Undangan Digital berskala global dengan performa transaksi yang cepat dan anti-down.
